[P1 封面]：  
標題建議：**Luiu 旅遊規劃網**  
副標題建議：**讓揪團出遊，從混亂討論變成清楚行程**  
頁面視覺意象建議：朋友一起看著手機規劃旅行，畫面中有地圖、景點照片與行程卡片，呈現「出發前就安心安排好」的感覺。

[P2 痛點]：

- 想出遊時，大家各自丟景點，資訊散在聊天紀錄裡，很難整理。
- 行程順序、交通方式、每天去哪裡，常常越討論越混亂。
- 旅行結束後，照片與回憶分散保存，想回顧或分享不夠方便。

[P3 解決方案]：  
**Luiu 把「找景點、排路線、整理行程、留下回憶」集中在同一個網站，讓旅行從規劃到分享都更順。**

[P4 使用場景]：  
小組要一起去高雄玩，成員先建立一份行程，加入想去的景點，安排每天的順序與交通方式；討論時可以直接看行程內容，不用在訊息裡翻來翻去。出遊後，也能把照片與旅遊故事整理成回憶，方便之後再次查看與分享。

再次條列核心功能：**行程列表管理、行程內容編輯、景點搜尋與地圖呈現、交通路線規劃、行程留言討論、旅遊回憶分享**。

**P5 行程列表管理：讓每趟旅行都一眼找到**
內容重點：

- 使用者可以看到自己建立的旅行計畫。
- 每張行程卡片呈現名稱、日期、封面與狀態。
- 可快速進入編輯或整理行程。

圖片擺放建議：右側放「我的行程」列表頁截圖，左側放重點文字。

**P6 直覺式行程編輯：讓每天去哪裡變得清清楚楚**
內容重點：

- 使用者可依天數整理景點安排。
- 景點順序清楚呈現，方便討論與調整。
- 每個景點可搭配停留資訊與備註。

圖片擺放建議：放一張行程編輯頁截圖，框出每日行程區塊。

**P7 景點搜尋與地圖：讓想去的地方立刻進入行程**
內容重點：

- 使用者可以在地圖上搜尋景點。
- 景點位置直接呈現在畫面中。
- 規劃時不用在多個網站之間切換。

圖片擺放建議：放地圖搜尋畫面截圖或 GIF，顯示搜尋景點到加入行程的流程。

**P8 交通路線規劃：讓行程不只好看，也真的走得順**
內容重點：

- 可依景點順序查看移動路線。
- 支援不同交通方式的安排。
- 幫助使用者判斷行程是否太趕或太分散。

圖片擺放建議：放地圖路線畫面截圖/GIF，路線線條要明顯。

**P9 行程留言討論：讓揪團意見集中在同一份計畫裡**
內容重點：

- 成員可以針對行程留下討論內容。
- 減少訊息散落在聊天軟體的混亂。
- 讓決策與行程內容放在同一個地方。

圖片擺放建議：放行程頁留言區截圖，建議用標註圈出留言欄位。

**P10 旅遊回憶分享：讓旅行結束後也能被好好收藏**
內容重點：

- 使用者可以建立旅遊回憶。
- 可整理旅行照片、天數與景點內容。
- 讓旅行從規劃延伸到回顧與分享。

圖片擺放建議：放回憶列表或回憶詳情頁截圖，畫面以照片與故事感為主。

**P11 技術牆：我們把複雜技術變成穩定的旅遊體驗**  
我們的網站不是單一頁面，而是一套完整系統：前台負責讓使用者看得懂、點得順；後台負責記住資料、處理規則、串接地圖服務。  
可以把它想成一間旅行社：前台接待親切好用，後方團隊則負責確認資料、安排路線、保存紀錄。  
這樣的分工讓使用者只感受到「簡單」，但背後其實有完整架構支撐。

**P12 前端亮點：讓規劃行程像操作手機 App 一樣直覺**  
前端畫面使用 Vue.js 與 Bootstrap 打造，讓頁面切換、行程卡片、地圖操作都更流暢。  
我們把畫面拆成許多可重複使用的小零件，像積木一樣組裝，讓團隊能更快完成不同頁面。  
對使用者來說，最大的好處是：找行程、編輯景點、查看地圖時，不需要學習成本。

**P13 後端與資料防護：像網站的大腦與保險箱**  
後端負責處理會員、行程、景點、路線與回憶等核心規則，就像網站的大腦，確保每個操作都有正確回應。  
資料保存則像保險箱，幫使用者把行程、景點、留言與回憶妥善整理起來。  
我們也設計了「軟刪除」概念，像資料防護網，避免重要內容被誤刪後完全消失。

**P14 團隊協作：用 Git 讓多人開發不互相踩到**  
我們使用 Git 進行分工，每位成員負責不同功能，像多人一起編輯同一本企劃書，但每個人都有自己的工作區。  
完成後再把成果合併，讓行程、會員、地圖、回憶等功能能逐步整合成完整網站。  
這種方式讓進度更清楚，也方便追蹤每個功能是誰完成、什麼時候完成。

**P15 衝突解決：遇到問題先溝通，再一起修正**  
多人合作時，難免會同時修改到相近的地方，我們把這稱為「版本衝突」。  
遇到衝突時，我們不是直接覆蓋彼此成果，而是先確認雙方修改目的，再討論保留哪一段、如何整合。  
這展現我們不只會寫功能，也能用理性溝通維持團隊效率與作品品質。

**P16 成果總結：在有限時間內完成一個可操作的旅遊規劃網站**  
文案：  
我們完成的不只是畫面展示，而是一個能從「建立行程」一路延伸到「安排景點、查看路線、留下回憶」的網站。  
使用者可以把原本散落在聊天紀錄、地圖、相簿裡的旅遊資訊，集中整理在同一個地方。  
這次成果展現的是團隊把想法落地、整合功能、完成產品流程的執行力。

**P17 技術挑戰：前後端資料串接與同步，讓畫面和資料說同一種語言**  
文案：  
遇到的問題：前端畫面需要行程、景點、交通、留言等資料，但不同功能使用的資料格式不完全一致，導致畫面有時無法正確顯示。  
我們如何處理：前端同學確認畫面需要哪些欄位，後端同學對照資料來源與回傳內容，大家一起看錯誤訊息與紀錄，逐步找出不一致的地方。  
最後的解法：統一整理資料欄位與傳遞方式，讓行程列表、行程內容、景點資訊與路線結果能更穩定地接上。  
學到的事：功能能不能順利運作，不只看單一頁面做得漂不漂亮，更重要的是團隊要把「資料怎麼流動」講清楚。

**P18 產品挑戰：把複雜的旅遊規劃，變成使用者看得懂的流程**  
文案：  
旅遊規劃本身很複雜，包含日期、景點、順序、交通、討論與回憶。  
我們的挑戰是不能把複雜度直接丟給使用者，而是要整理成清楚的行程卡片、每日安排與地圖畫面。  
這讓我們學會從使用者角度思考：不是功能越多越好，而是每一步都要讓人知道自己接下來該做什麼。

**P19 團隊收穫：從各做各的功能，走向真正的協作產品**  
文案：  
一開始大家負責不同區塊，例如會員、行程、地圖、回憶與畫面設計。  
到整合階段時，我們必須頻繁確認彼此的進度與需求，避免功能各自完成卻接不起來。  
這次經驗讓我們理解，團隊開發不是把工作切開就結束，而是要持續溝通、對齊規格，最後一起完成同一個產品。

未來可以持續優化操作細節，讓共同規劃更即時、行程調整更順手、回憶分享更完整。
